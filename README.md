REST API для управления организационной структурой компании.

## Технологии

* Go
* net/http
* PostgreSQL
* GORM
* Goose
* Docker
* Docker Compose
* httptest

---

# Архитектура

Проект построен по слоистой архитектуре:

```text
Handler -> Service -> Repository -> PostgreSQL
```

### Handler

Обрабатывает HTTP-запросы и формирует HTTP-ответы.

### Service

Содержит бизнес-логику приложения:

* проверка уникальности подразделений;
* защита от циклических зависимостей;
* построение дерева подразделений;
* обработка сценариев удаления подразделений.

### Repository

Работа с базой данных через GORM.

### Database

PostgreSQL в качестве основного хранилища данных.

# Возможности

## Подразделения

* Создание подразделения
* Получение подразделения с вложенной структурой подразделений
* Изменение подразделения
* Удаление подразделения

Дополнительно:

* поддержка иерархии подразделений;
* защита от циклических ссылок;
* уникальность имени подразделения в рамках одного родителя;
* получение дерева подразделений с ограничением глубины вложенности;
* каскадное удаление дочерних подразделений;
* перенос сотрудников при удалении подразделения.

## Сотрудники

* Создание сотрудника
* Получение сотрудников подразделения по желанию

# Запуск проекта

## Сборка и запуск

```bash
docker compose up --build
```

При запуске автоматически:

1. Поднимается PostgreSQL.
2. Выполняются миграции Goose.
3. Запускается API.

После запуска API будет доступно по адресу:

```text
http://localhost:8080
```

# API

## Создать подразделение

```http
POST /departments
```

Request:

```json
{
  "name": "IT",
  "parent_id": null
}
```

Response:

```json
{
  "id": 1,
  "name": "IT",
  "parent_id": null,
  "created_at": "2026-06-03T12:00:00Z"
}
```

---

## Получить подразделение

```http
GET /departments/{id}
```

Query параметры:

| Параметр          | По умолчанию | Описание                                       |
| ----------------- | ------------ | ---------------------------------------------- |
| depth             | 1            | Глубина вложенности подразделений (максимум 5) |
| include_employees | true         | Включать сотрудников подразделения             |

Пример:

```http
GET /departments/1?depth=3&include_employees=true
```

```json
{
  "id": 1,
  "name": "IT",
  "children": [
    {
      "id": 2,
      "name": "Backend",
      "children": [
        {
          "id": 3,
          "name": "Go Team"
        }
      ]
    }
  ]
}
```

---


## Обновить подразделение

```http
PATCH /departments/{id}
```

Request:

```json
{
  "name": "Development",
  "parent_id": 2
}
```

---

## Удалить подразделение

```http
DELETE /departments/{id}
```

Query параметры:

| Параметр    | Описание                                  |
| ----------- | ----------------------------------------- |
| mode        | cascade или reassign                      |
| reassign_to_department_id | ID подразделения для переноса сотрудников |

Пример:

```http
DELETE /departments/3?mode=reassign&reassign_to_department_id=1
```

---

## Создать сотрудника

```http
POST /departments/{id}/employees
```

Где `{id}` — идентификатор подразделения.

Request:

```json
{
  "full_name": "Ivan Ivanov",
  "position": "Backend Developer",
  "hired_at": "2025-01-15T00:00:00Z"
}
```

# Бизнес-правила

* Нельзя создать подразделение с пустым именем.
* Имя подразделения должно быть уникальным в рамках одного родительского подразделения.
* Подразделение не может быть родителем самого себя.
* Запрещено создавать циклические связи в дереве подразделений.
* Нельзя создать сотрудника в несуществующем подразделении.
* Максимальная глубина вложенности при получении дерева подразделений — 5.
* При удалении подразделения поддерживаются режимы:

  * `cascade` — удалить подразделение вместе со всеми дочерними подразделениями;
  * `reassign` — перенести сотрудников в другое подразделение.

# Логирование

Для логирования HTTP-запросов используется middleware.

Формат лога:

```text
METHOD PATH STATUS DURATION
```

Пример:

```text
GET /departments/1 200 1.2ms
POST /departments 201 2.4ms
PATCH /departments/1 200 1.1ms
```

# Особенности реализации

* Используется стандартный пакет net/http.
* Маршруты реализованы через HTTP ServeMux из Go 1.22+.
* Работа с БД реализована через GORM.
* Миграции выполняются через Goose.
* Поддерживается Docker и Docker Compose.
* Реализована защита от циклических связей подразделений.
* Реализована загрузка дерева подразделений по уровням (BFS), что уменьшает количество запросов к БД.
* Уникальность контролируется как на уровне бизнес-логики, так и на уровне базы данных через уникальный индекс.
* Используется middleware для логирования запросов.
* Код разделён на слои Handler / Service / Repository.
